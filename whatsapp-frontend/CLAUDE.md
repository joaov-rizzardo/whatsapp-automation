# This is NOT the Next.js you know

# whatsapp-frontend

Stack: Next.js 16.2 (App Router) · React 19.2 · TypeScript strict · Tailwind v4 · React Query (client-side fetching) · React Hook Form + Zod (forms).

**We do not write unit tests in this project.** Validate changes by running the app (`npm run dev`), not by writing tests.

Import alias: `@/*` resolves to the project root (e.g. `@/lib/http`).

## Next 16 — what changed since your training data

These are the traps that come up most. When unsure, read `node_modules/next/dist/docs/01-app/` before writing.

**Request APIs are async.** Synchronous access was removed in 16 — there is no fallback. `cookies()`, `headers()`, `draftMode()`, `params`, and `searchParams` are Promises:

```tsx
// app/conversations/[id]/page.tsx
export default async function Page(props: PageProps<'/conversations/[id]'>) {
  const { id } = await props.params
  const filters = await props.searchParams
}
```

`PageProps` / `LayoutProps` / `RouteContext` are globals generated by `npx next typegen`. Prefer them over typing `params` by hand.

**`fetch` is no longer cached by default.** It blocks rendering until it resolves. For fresh data on every request, wrap the component in `<Suspense>` and let it stream. To cache, use the `use cache` directive.

**`middleware.ts` is now `proxy.ts`** at the root, with the function exported as `proxy`. The runtime is `nodejs` and is not configurable — `edge` is not supported in `proxy`. Config flags were renamed to match (`skipMiddlewareUrlNormalize` → `skipProxyUrlNormalize`).

**Cache Components** (`cacheComponents: true` in `next.config.ts`) is how you opt into PPR in 16 — the `experimental.ppr` flag and the `experimental_ppr` segment export were removed. It is **not** enabled in this project today; don't enable it without agreeing first, since it changes the caching model for the whole app.

**Cache APIs:**
- `revalidateTag(tag, profile)` — the second argument (a `cacheLife` profile, e.g. `'max'`) is now required; the single-argument form is a type error. Stale-while-revalidate semantics.
- `updateTag(tag)` — Server Actions only; expires and refreshes within the same request (read-your-writes). This is the right call when the user must see their own change immediately after submitting a form.
- `refresh()` — refreshes the client router from within a Server Action.
- `cacheLife` / `cacheTag` are stable: import them without the `unstable_` prefix.

**`next lint` was removed.** Linting is `npm run lint` (ESLint directly, flat config in `eslint.config.mjs`). `next build` no longer runs lint.

**Turbopack is the default** for both `dev` and `build`.

**`next/image`:** `images.domains` is deprecated — use `images.remotePatterns`. `next/legacy/image` was removed.

## Folder structure

`app/` is for routing only. Everything else lives outside it, grouped by feature.

```
app/                          # routes only: page/layout/loading/error/route
  conversations/[id]/page.tsx
  layout.tsx
  providers.tsx               # client providers mounted in RootLayout
features/                     # one folder per domain
  conversations/
    components/               # feature-specific UI
    hooks/                    # feature logic (queries, mutations, state)
    api/                      # HTTP calls for this feature
    schemas/                  # Zod schemas
    types/                    # types not derived from a schema
components/                   # cross-feature, presentational UI
  ui/                         # base elements: Button, Input, Dialog
hooks/                        # cross-feature hooks
lib/                          # infrastructure: http client, utils, constants
types/                        # global/shared types
```

**Every file goes in the folder that matches its role.** A hook belongs in `hooks/`, a schema in `schemas/`, an HTTP call in `api/` — never inline in the component file that happens to use it. If a file's location doesn't tell you what's inside it, it's in the wrong folder.

Start in `features/<domain>/`. Promote to the root-level folders (`components/`, `hooks/`, `lib/`) only once something is actually used by a second feature — don't pre-generalize.

Route files (`page.tsx`, `layout.tsx`) stay thin: they compose feature components and handle server-side data fetching. Business logic does not live in `app/`.

## One file, one purpose

Each file exports one thing and has one reason to change: one component, one hook, one schema, one API call. The filename says what that thing is; the export matches the filename.

Don't co-locate a component, its hook, its schema, and its fetch call in one file because they're related — related is what the feature folder is for. Barrel files (`index.ts`) that re-export are fine; files that *define* several unrelated things are not.

## No logic in visual components

**Visual components render — they don't decide.** A component under `components/` receives data and callbacks as props, and returns JSX. It does not fetch, transform data, hold business rules, or reach into React Query.

All of that goes in a **custom hook**. The hook owns fetching, mutations, derived state, and side effects; the component consumes its return value:

```tsx
// features/conversations/hooks/useConversation.ts
export function useConversation(id: string) {
  const { data, isLoading } = useQuery({
    queryKey: ['conversations', id],
    queryFn: () => fetchConversation(id),
  })

  return {
    conversation: data,
    isLoading,
    unreadCount: data?.messages.filter((m) => !m.read).length ?? 0,
  }
}
```

```tsx
// features/conversations/components/ConversationPanel.tsx
'use client'

export function ConversationPanel({ id }: { id: string }) {
  const { conversation, isLoading, unreadCount } = useConversation(id)

  if (isLoading) return <Spinner />
  return <ConversationView conversation={conversation} unread={unreadCount} />
}
```

Rules of thumb: no `useQuery`/`useMutation` inside a presentational component; no `.filter()`/`.map()` computing business meaning in JSX; no conditionals encoding business rules inline. If a component is hard to read at a glance, the logic in it wants to be a hook.

Container components (the ones calling the hook) are the exception — they may hold the hook call, but they still delegate rendering downward.

## Server and Client Components

Everything is a Server Component by default. Only mark `'use client'` when you need state, event handlers, lifecycle, custom hooks, or browser APIs.

Push `'use client'` to the leaves of the tree. Fetch on the server and pass data as props into the interactive island — don't turn a whole page into a client component because of one button. Props crossing the boundary must be serializable.

Server Components can access secrets and the backend directly without leaking into the bundle. **Every Server Function is reachable via a direct POST, not just through your UI** — always verify authentication and authorization inside each one.

## Data fetching

The project rule: **Server Component when the data can be fetched on the server; React Query when the fetch happens on the client.**

React Query is for server state on the client — data that needs refetching, polling, a cache shared across components, or user interaction (for WhatsApp: a live message list, connection status). Don't use React Query for what a Server Component already solves, and never fetch with `useEffect` + `useState`.

Setup (none of this is installed yet — when installing, check the current pattern at https://tanstack.com/query/latest/docs/framework/react/guides/advanced-ssr):

- `QueryClientProvider` lives in a Client Component (`app/providers.tsx`) mounted in `RootLayout`. The provider is a client component; the layout stays a server component.
- **Never share a `QueryClient` across requests.** On the server, create a new one per request; in the browser, use a singleton (this avoids recreating it if React suspends on the first render).
- Set `staleTime` above 0 (e.g. `60 * 1000`) — without it the client refetches immediately after hydration.
- To prefetch on the server: `prefetchQuery` + `<HydrationBoundary state={dehydrate(queryClient)}>` in the Server Component, and `useQuery` with the **same query key** on the client.

Query keys describe the data and include every variable the fetch depends on (`['conversations', id]`), otherwise the cache serves another entity's data. Mutations invalidate via `queryClient.invalidateQueries`.

## Forms

React Hook Form + Zod via `zodResolver`, for every form. The Zod schema is the source of truth for the type — infer it, don't redeclare it:

```tsx
// features/conversations/schemas/sendMessage.ts
export const sendMessageSchema = z.object({
  phone: z.string().min(1, 'Phone is required'),
  message: z.string().min(1, 'Message is required'),
})

export type SendMessageInput = z.infer<typeof sendMessageSchema>
```

```tsx
// features/conversations/components/SendMessageForm.tsx
'use client'

export function SendMessageForm() {
  const { register, handleSubmit, onSubmit, errors, isSubmitting } = useSendMessage()

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('phone')} />
      {errors.phone && <span>{errors.phone.message}</span>}
      <button disabled={isSubmitting}>Send</button>
    </form>
  )
}
```

- The schema lives in `schemas/`, the `useForm` call and submit handler in a hook under `hooks/` — the component only wires them to inputs.
- When schema input and output diverge (coercion, transform), type it explicitly: `useForm<z.input<typeof s>, any, z.output<typeof s>>`.
- Numeric fields: `register('age', { valueAsNumber: true })`.
- Re-validate on the server with the same schema. Client validation is UX, not security.
- A submit that calls the API from the client → React Query's `useMutation`, inside the hook.
- If the form is a Server Action with `<form action={...}>`, the pairing is `useActionState` (not RHF) — the Server Function then receives `prevState` as its first argument.

## Tailwind v4

**There is no `tailwind.config.js` in this project — do not create one.** v4 is CSS-first: config lives in `app/globals.css`, via `@import "tailwindcss"` and the `@theme inline` block. New tokens (colors, fonts, spacing) go there as custom properties — a `--color-*` defined in `@theme` becomes a utility (`bg-background`, `text-foreground`).

Style with utilities in the JSX. Don't write loose CSS or CSS Modules for what a utility solves; `globals.css` is for theme tokens and resets, not component classes. Repeated class strings are solved by extracting a **React component**, not `@apply`.

Class order matters for conflicts: use `clsx`/`tailwind-merge` for conditional classes instead of concatenating strings.

## Conventions

- Route files in `app/`, following the file conventions (`page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, `route.ts`).
- `error.tsx` and `loading.tsx` per route segment instead of handling error/loading states by hand in the page.
- `loading.tsx` or `<Suspense>` for any data that might be slow — don't block the whole route.
- No implicit `any`; `tsconfig` is `strict`. Derive types from Zod schemas wherever a schema exists.
